Stack vs Heap — Chi tiết kỹ thuật

Phân biệt hai vùng nhớ: Stack — quản lý bởi compiler/runtime, LIFO; Heap — quản lý bởi GC.

Stack Frame Layout

Mỗi frame: Local Variable Array — slot 32-bit (long/double dùng 2 slot); Operand Stack — max depth xác định tại compile; Reference tới Constant Pool của method. Kích thước frame = tổng local vars + operand stack + overhead. -Xss256k giới hạn stack/thread — thread nhiều = stack nhiều.

Phân bổ biến — Ví dụ

void method() {
  int a = 1;              // a: primitive trên Stack
  Integer b = Integer.valueOf(1);  // b: reference Stack; Integer: Heap (cached)
  Object o = new Object();  // o: reference Stack; Object: Heap
  int[] arr = new int[10];  // arr: reference Stack; array: Heap
}
// Return: frame pop, a/b/o/arr mất; Object/array chờ GC

Escape Analysis

JIT phân tích Object có "escape" khỏi method không. No escape: Object chỉ dùng nội bộ method → Scalar Replacement: allocate trên stack thay vì Heap, hoặc bỏ allocation (chỉ lưu giá trị trong register). Giảm áp lực GC. Bật mặc định: -XX:+DoEscapeAnalysis.

// Object không escape — có thể scalar replace
void sum() {
  Point p = new Point(1, 2);  // Chỉ dùng trong method
  return p.x + p.y;  // JIT có thể bỏ allocation
}

StackOverflowError

Xảy ra khi: đệ quy quá sâu; frame quá lớn (nhiều local variable); circular call. Giải pháp: tăng -Xss; refactor đệ quy thành vòng lặp.

So sánh Stack và Heap
Đặc điểm Stack Heap
Đối tượng Primitive, reference, Frame Object (header + fields)
Allocation Push frame — O(1) Bump pointer (TLAB) hoặc free list
Deallocation Pop frame — ngay khi return GC thu hồi (bất đồng bộ)
Fragmentation Không Có — cần compact
Lỗi StackOverflowError OutOfMemoryError: Java heap space
