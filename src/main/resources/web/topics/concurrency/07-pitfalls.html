Các lỗi kinh điển cần tránh

Deadlock

Thread A giữ Lock 1 đợi Lock 2; Thread B giữ Lock 2 đợi Lock 1. Cả hai đứng mãi. Cách tránh: Luôn lấy lock theo thứ tự cố định (A rồi B); dùng tryLock với timeout.

// Ví dụ deadlock
Object lock1 = new Object();
Object lock2 = new Object();

Thread t1 = new Thread(() -> {
  synchronized (lock1) {
    try { Thread.sleep(10); } catch (InterruptedException e) { }
    synchronized (lock2) { }  // Đợi lock2
  }
});
Thread t2 = new Thread(() -> {
  synchronized (lock2) {
    try { Thread.sleep(10); } catch (InterruptedException e) { }
    synchronized (lock1) { }  // Đợi lock1 — DEADLOCK
  }
});

// Tránh: luôn lấy lock1 trước lock2
synchronized (lock1) {
  synchronized (lock2) {
    // ...
  }
}

Livelock

Hai thread liên tục nhường nhịn nhau, không việc gì hoàn thành. Ví dụ: hai người qua cửa, cả hai lùi rồi tiến lại. Cách tránh: Thêm randomness hoặc backoff.

Starvation

Thread "đói" tài nguyên vì thread ưu tiên cao chiếm hết CPU. Cách tránh: Dùng ReentrantLock(true) — fair lock; cân bằng priority.

Race Condition

Kết quả phụ thuộc thứ tự thực thi — thread nào chạy trước. Ví dụ: i++ không atomic (read-modify-write). Cách tránh: synchronized, AtomicInteger, volatile (chỉ visibility).

Memory Visibility

Thread A ghi biến; Thread B có thể đọc giá trị cũ từ CPU cache. volatile đảm bảo ghi visible cho mọi thread. synchronized cũng tạo memory barrier. Happens-before: ghi trong synchronized block visible cho đọc trong block sau khi cùng lock.

// SAI — không volatile, B có thể không thấy flag = true
boolean ready = false;
// Thread A: ready = true;
// Thread B: while (!ready) {} — có thể loop vô hạn

// Đúng
volatile boolean ready = false;

Double-Checked Locking (sai)

Pattern singleton với lazy init có thể sai nếu không dùng volatile. Từ Java 5+, volatile fix vấn đề. Tốt hơn: dùng Holder class hoặc enum singleton.

// Sai (trước Java 5) — có thể thấy object chưa init xong
private static Singleton instance;
public static Singleton getInstance() {
  if (instance == null) {
    synchronized (Singleton.class) {
      if (instance == null) {
        instance = new Singleton();  // Có thể reorder
      }
    }
  }
  return instance;
}
// Đúng: private static volatile Singleton instance;

Lời khuyên

Ưu tiên: Dùng class có sẵn trong java.util.concurrent thay vì tự viết cơ chế khóa bằng synchronized/wait-notify. Dùng ExecutorService, ConcurrentHashMap, BlockingQueue.
Tránh: Tự implement lock phức tạp; dùng synchronized khi có thể dùng Atomic*; quên shutdown ExecutorService.
